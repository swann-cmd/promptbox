## 🎯 完整工作流（简化版）

```
┌─────────────────────────────────────────────────────────────────┐
│                     PromptBox 开发工作流                          │
└─────────────────────────────────────────────────────────────────┘

1️⃣  计划阶段
┌─────────┐   ┌─────────┐   ┌─────────┐
│产品想法  │ → │设计方案  │ → │技术方案  │
└─────────┘   └─────────┘   └─────────┘
                   ↓
         使用 /office-hours 重新构建

2️⃣  开发阶段
┌─────────┐   ┌─────────┐   ┌─────────┐
│编写代码  │ → │代码检查  │ → │质量审核  │
└─────────┘   └─────────┘   └─────────┘
                   ↓
         改动 > 200 行 → 先简化

3️⃣  测试阶段
┌─────────┐   ┌─────────┐   ┌─────────┐
│单元测试  │ → │集成测试  │ → │E2E测试  │
│npm test │   │  /qa    │   │  /qa    │
└─────────┘   └─────────┘   └─────────┘
                   ↓
         测试失败 → /investigate 调试

4️⃣  审查阶段
┌─────────┐   ┌─────────┐   ┌─────────┐
│提交代码  │ → │创建 PR  │ → │代码审查  │
│git push │   │  /ship  │   │ /review │
└─────────┘   └─────────┘   └─────────┘
                   ↓
         发现问题 → 修复后重新提交

5️⃣  部署阶段
┌─────────┐   ┌─────────┐   ┌─────────┐
│合并 PR  │ → │部署生产  │ → │监控健康  │
│/ship    │   │/deploy  │   │/canary  │
└─────────┘   └─────────┘   └─────────┘
                   ↓
         发现问题 → 热修复或回滚
```

## 🚀 每日开发工作流

```
早上开始工作：
├─ 1. 拉取最新代码
│   git pull
│
├─ 2. 切换到功能分支
│   git checkout -b feature/my-feature
│
├─ 3. 开发中保持质量
│   npm test          # 保持测试通过
│   npm run lint      # 检查代码规范
│
├─ 4. 提交前检查
│   git diff --stat   # 查看改动量
│   # 如果 > 200 行，考虑先简化
│
├─ 5. 提交代码
│   git add .
│   git commit -m "feat: 清晰的提交信息"
│
├─ 6. 审核和测试
│   npm test
│   # 如涉及关键代码，运行 /review
│
└─ 7. 推送代码
    git push origin feature/my-feature

中午/下午：
├─ 8. 创建 PR
│   /ship
│
├─ 9. 代码审查
│   /review (自动审核)
│   等待团队审查
│
└─ 10. 修改反馈
    修复审查意见
    更新 PR

结束工作前：
├─ 11. 部署到生产
│   /land-and-deploy
│
├─ 12. 监控健康
│   /canary
│
└─ 13. 更新文档
    /document-release
```

## 📊 质量指标速查表

### **代码质量**

```yaml
组件规模:
  ✅ 推荐: < 300 行
  ⚠️  警告: 300-500 行
  ❌ 禁止: > 500 行

函数规模:
  ✅ 推荐: < 50 行
  ⚠️  警告: 50-100 行
  ❌ 禁止: > 100 行

嵌套层级:
  ✅ 推荐: ≤ 3 层
  ⚠️  警告: 4-5 层
  ❌ 禁止: > 5 层

参数数量:
  ✅ 推荐: ≤ 3 个
  ⚠️  警告: 4-5 个
  ❌ 禁止: > 5 个（考虑封装对象）
```

### **测试覆盖**

```yaml
关键路径:
  ✅ 要求: 100% 覆盖
  • 用户认证
  • 支付流程
  • 权限控制

重要功能:
  ✅ 目标: > 80% 覆盖
  • 核心业务逻辑
  • API 调用
  • 状态管理

UI 组件:
  ✅ 目标: > 70% 覆盖
  • 交互逻辑
  • 表单验证
```

### **性能指标**

```yaml
加载时间:
  ✅ 目标: < 2.5s
  ⚠️  警告: 2.5-4s
  ❌ 禁止: > 4s

响应时间:
  ✅ 目标: < 200ms
  ⚠️  警告: 200-500ms
  ❌ 禁止: > 500ms

错误率:
  ✅ 目标: < 0.1%
  ⚠️  警告: 0.1-0.5%
  ❌ 禁止: > 0.5%
```

## 🛠️ 工具快速参考

### **开发工具**

```bash
# 包管理
npm install          # 安装依赖
npm run dev          # 启动开发服务器

# 代码质量
npm run lint         # 检查代码规范
npm run format       # 格式化代码

# 测试
npm test             # 运行所有测试
npm test -- -u       # 运行特定测试
npm test -- --coverage  # 查看测试覆盖率
```

### **Git 工作流**

```bash
# 分支管理
git branch           # 查看分支
git checkout -b feature/name  # 创建功能分支
git checkout main    # 切换到主分支

# 提交流程
git add .            # 添加所有更改
git status          # 查看状态
git diff            # 查看改动
git commit -m "msg"  # 提交更改

# 推送流程
git push origin branch-name  # 推送分支
git pull            # 拉取最新代码
```

### **gstack 技能**

```bash
# 规划阶段
/office-hours       # 重新构建产品
/plan-eng-review    # 技术方案审核
/plan-design-review # 设计方案审核

# 质量保证
/review            # 自动代码审核
/simplify          # 代码简化建议
/qa                # 端到端测试
/investigate       # 根因调试

# 部署流程
/ship              # 创建 PR 并推送
/land-and-deploy   # 合并并部署
/canary            # 部署后监控
/document-release  # 更新文档
```

## 📝 提交信息规范

```bash
# 格式
<type>(<scope>): <subject>

# 类型 (type)
feat:     新功能
fix:      Bug 修复
refactor: 重构
docs:     文档
style:    格式
test:     测试
chore:    构建/工具

# 示例
feat(auth): 添加 GitHub OAuth 登录
fix(payment): 修复支付金额计算错误
refactor(ui): 提取可复用按钮组件
docs(api): 更新 API 文档
```

## 🚨 紧急情况处理

### **发现生产 Bug**

```bash
# 1. 立即回滚
git revert HEAD
git push

# 2. 或在 Vercel 控制台
# Deployments → 找到稳定版本 → Promote to Production
```

### **紧急热修复**

```bash
# 1. 创建 hotfix 分支
git checkout -b hotfix/critical-bug

# 2. 快速修复
# 修复代码...

# 3. 跳过测试直接推送（如果确实紧急）
git add .
git commit -m "fix: 修复关键生产 bug"
git push origin hotfix/critical-bug

# 4. 快速合并
git checkout main
git merge hotfix/critical-bug
git push
```
